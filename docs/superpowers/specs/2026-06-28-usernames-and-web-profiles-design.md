# Claimable Usernames + Web Profiles — Design Spec

**Date:** 2026-06-28
**Status:** Draft for review
**Repos:** `Seasons_AIv02` (mobile signup + claim UX), `seasonz-dashboard` (Next.js — public web profiles), shared Supabase (Tokyo)

## Goal

Give every user a **claimable username** (`@nathaniel` instead of the auto-generated `@nathaniel.9432`), then expose **shareable public web profiles** at `seasonz.ai/@handle` that honour the user's existing season-visibility settings.

Two phases, each independently shippable (recommend **one implementation plan per phase**):
- **Phase 1 — Claimable handles** (incl. signup impact). Build now.
- **Phase 2 — Web profiles.** Build after Phase 1.

## Current state (verified)

- `profiles.handle text unique`. Auto-generated by the `handle_new_user()` trigger: it slugifies the signup `display_name` and calls `generate_unique_handle()` which appends a random 4-digit suffix (`nathaniel.9432`). Handles therefore already use lowercase letters, digits and `.`.
- Signup: `SignUpScreen` collects `name, email, password` → account created → `OnboardingScreen` runs. No handle-claim step exists; the user never chooses their handle.
- Season visibility already exists per season: `open` / `connections` / `private` (the profile privacy picker). Public web profiles will show only `open` content.

## Principles

- **The auto-generated handle stays as a safe default.** Claiming is an upgrade, never a blocker — a user who skips still has a working unique handle.
- **All handle writes go through one validated RPC** (`claim_handle`) — format, reserved-list, uniqueness, and change-cooldown enforced server-side. No client-side handle writes.
- **Public web profiles never leak more than the season-visibility settings allow** — only `open` content is exposed to anonymous viewers.

# Phase 1 — Claimable Handles

## Handle rules

- 3–20 characters, **lowercase** `a–z`, digits, `.` and `_`.
- Must **start with a letter**.
- No leading/trailing separator, no consecutive separators (`..`, `__`, `._`).
- Stored lowercased; uniqueness is case-insensitive.
- Not in the reserved list.

Regex (used in the RPC): `^[a-z][a-z0-9._]{2,19}$`, plus rejects matching `[._]{2,}` or `[._]$`.

## Data model

### `reserved_handles` (new)
```sql
create table if not exists public.reserved_handles (label text primary key);
insert into public.reserved_handles (label) values
  ('admin'),('administrator'),('support'),('help'),('seasonz'),('seasons'),
  ('official'),('team'),('mod'),('moderator'),('root'),('api'),('www'),
  ('about'),('privacy'),('terms'),('contact'),('login'),('signup'),('settings'),
  ('me'),('user'),('null'),('undefined')
on conflict do nothing;
-- Readable by no one client-side; only the definer RPCs consult it.
alter table public.reserved_handles enable row level security;
```

### `profiles` — change cooldown (ALTER)
```sql
alter table public.profiles
  add column if not exists handle_changed_at timestamptz;
```

## RPCs

### `check_handle(p_handle text) returns jsonb` — live availability (no side effects)
Used by the claim UI for debounced real-time feedback. Returns `{ available: boolean, reason: 'ok'|'invalid_format'|'reserved'|'taken' }`. Ignores cooldown (that's only enforced on commit). SECURITY DEFINER, `set search_path = public`, granted to authenticated.

### `claim_handle(p_handle text) returns jsonb` — validate + set
1. `auth.uid()` required.
2. Normalise (`lower(trim())`), check format → `invalid_format`.
3. Reserved list → `reserved`.
4. Taken by another user (case-insensitive) → `taken`.
5. **Cooldown** (30 days) since `handle_changed_at`, unless the new handle equals the current one → `cooldown` (+ `next_at`).
6. `update profiles set handle = <clean>, handle_changed_at = now()`.
7. Return `{ ok: true, handle }` or `{ ok: false, reason, next_at? }`.
SECURITY DEFINER, `set search_path = public`, granted to authenticated.

This RPC is reused by **both** the signup claim step and the later "edit username" settings UI.

## Signup / onboarding impact

The account (and a default suffixed handle) exists by the time onboarding runs, so claiming happens with an authenticated session.

- **New onboarding step: "Choose your username"** — added as the first step of `OnboardingScreen` (the signup-completion flow).
  - Pre-fills the input with the slug of the user's name (e.g. `nathaniel`), shown with the `@` prefix.
  - **Live availability** via debounced `check_handle` — green "available" / red reason as they type.
  - **Confirm** calls `claim_handle`; on `ok` proceeds. On failure, shows the reason inline.
  - **Skip** is allowed → keeps the auto-generated default; the user can claim later in settings.
- `SignUpScreen` is unchanged (still name/email/password); the username choice is deliberately after account creation so live-checking and the authenticated `claim_handle` work cleanly. (Optionally, `SignUpScreen` can *preview* a suggested `@handle` under the name field, but the real claim is the onboarding step.)
- The `handle_new_user()` trigger is unchanged — it still produces the default.

## Edit-username (settings)

A "username" row in the profile/settings area (mobile `ProfilePanel` / profile edit) opens a small editor reusing the same live-check + `claim_handle` flow, surfacing the 30-day cooldown message when applicable. New `useClaimHandle` mutation + `useCheckHandle` query wrap the RPCs.

## Phase 1 testing (Vitest, pure logic)

- RPCs/onboarding verified by running the flow.

# Phase 2 — Web Profiles (`seasonz.ai/@handle`)

## Approach

Public, unauthenticated read-only profile pages on the Next.js web app (the `seasonz-dashboard` repo), served at `seasonz.ai/@handle`.

- **Routing:** internal route `app/u/[handle]/page.tsx`; a middleware rewrite maps `/@<handle>` → `/u/<handle>` so the public-facing URL is `seasonz.ai/@handle`. (Next.js can't use a literal `@` folder — `@` denotes parallel routes — hence the rewrite.)
- **Data:** a SECURITY DEFINER RPC `get_public_profile(p_handle text) returns jsonb` returning only public-safe fields: display name, handle, avatar, bio, and the user's **`open`-visibility** seasons/tags/milestones. Returns null if not found. This is the *only* anon-reachable path; it never returns `connections`/`private` content. Granted to the `anon` role.
- **Rendering:** server component reads `get_public_profile`, renders a clean profile (seasons as the three-season layout, tags, public milestones). 404 for unknown / no-public-content handles as appropriate.
- **SEO / sharing:** per-profile `<title>`/OpenGraph tags (name + tagline) and an OG image so shared links preview well.

## Mobile impact (Phase 2)

- A **"Share profile"** affordance (in `ProfilePanel`) copying/sharing `https://seasonz.ai/@<handle>`.

## Phase 2 open items to confirm at plan time

- Exact source of per-season visibility (`open`/`connections`/`private`) to filter on in `get_public_profile` — confirm the column/table during planning.
- Whether an empty public profile (no `open` content) renders a minimal card or 404.
- OG image generation approach (static vs. dynamic).

## Cross-cutting decisions

| Question | Decision |
| --- | --- |
| Real `@seasonz.ai` email | **No** — out of scope; usernames are handles/URLs, not mailboxes. |
| Default handle | Keep the auto-generated suffixed handle as the fallback; claiming is optional. |
| Username changes | Allowed, **30-day cooldown**; old web links may 404 after a change (acceptable for v1; redirects are a future enhancement). |
| Reserved names | `reserved_handles` table, seeded; editable by DB admins. |
| Verification badge | Out of scope (future; clean handles make impersonation of notable people easier — revisit). |

## Out of scope (future)

- Real email identities.
- Handle-change redirects / handle history.
- Verified badges.
- Custom profile theming on web.
- Discovery/search by handle beyond existing user search.

## Migration & rollout notes

- Phase 1 migration applied live from `Seasons_AIv02` (Tokyo); copy to dashboard repo for record (same pattern as prior features).
- All existing users keep their current suffixed handle until they choose to claim; no backfill.
- Phase 2's `get_public_profile` grant to `anon` is the only place anonymous DB access is introduced — keep its returned shape strictly limited to public content and review carefully.
