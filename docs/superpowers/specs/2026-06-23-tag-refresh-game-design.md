# Tag Refresh Game — Design Spec

**Date:** 2026-06-23
**Status:** Approved for planning
**Spans two repos:** `Seasons_AIv02` (React Native mobile app — players) and `seasonz-dashboard` (Next.js dashboard — champions/admin authoring).

---

## 1. Summary

A falling-words mini-game that keeps members' profile tags fresh and current, played on
mobile and authored from the dashboard.

Words fall from the top of the screen (max two on screen at once). The player flicks each
word into one of four outcomes, mapping to the existing season tag model:

- **Flick up → Not relevant** — soft-dismissed: hidden from active display/discovery but kept in history (never hard-deleted).
- **Flick left → Past** — "used to be me."
- **Flick down → Present** — "me right now." A *deliberate* downward flick.
- **Flick right → Future** — "where I'm heading."
- **Untouched → Skipped** — a word that reaches the bottom edge without a flick is re-queued, never auto-assigned.

Champions build word banks (typed manually or generated by AI from a theme), set the speed
and audience, and push a round to members' feeds. Completing (or bailing out of) a round
produces a light summary that the player commits with **"Update my seasons"**, writing the
results to their profile tags — which in turn feeds discovery.

This is a new, playful surface for writing `user_tags`; it does not replace manual tag editing.

---

## 2. Goals & Non-Goals

### Goals
- Refresh members' past/present/future tags through play, producing more current, honest tag data.
- Give champions/admins a self-serve tool to build, push, and measure tag-refresh rounds.
- Feed discovery (people/content matching) with fresher tags.
- Emoji as an aesthetic display layer over real text tags.

### Non-Goals (explicitly out of scope for v1)
- **Leaderboards / competitive scoring.** Scoring is personal-only and low-pressure.
- **Scheduling / recurring pushes.** v1 is "send now." (The settings UI may show a disabled
  "schedule" affordance, but it is deferred.)
- **Per-word AI emoji re-generation.** AI suggests an emoji at bank-generation time; manual edits only after.
- **Pre-assigning words to seasons by the champion.** Word banks are season-agnostic by design —
  sorting is the whole point of the game.

## 3. Player Experience (Seasons_AIv02)

### 3.1 Entry point
When a round is pending for the user, a "Tag Refresh" card appears on the Discover / Home
present feed. Tapping it opens the game.

### 3.2 Playfield (chosen layout: "Option B — four flicks, fall = skip")
- Obsidian field (`#0a0e1a`), Inter type, per-season accent gutters with dashed boundaries so
  each region is unmistakable:
  - **Past** — left gutter, coral `#f87559`.
  - **Future** — right gutter, sky blue `#60a5fa`.
  - **Present** — bottom gutter, grass green `#22c55e`.
  - **Not relevant** — top gutter, neutral muted grey (`#5d6580`/`#9aa3b8`); no season owns it.
- The one-time instruction ("Where does this belong?") shows on a start card, then the field
  clears for play.
- **Slim top bar during play:** segmented progress bar; left = `done/total` count + a small
  glowing speed dot; right = streak (`🔥 ×N`) and score.
- Max two word-chips on screen: a lead chip near center plus a dimmed "next" chip queued above.
- A word reaching the bottom untouched is **skipped and re-queued** — never auto-tagged as Present.

### 3.3 Word chips (emoji = display layer)
Each bank word renders in one of three `display_mode`s, chosen per word by the champion/AI:
- **text** — e.g. `Mentoring` (concept words with no obvious icon).
- **combo** — e.g. `✈️ Travel` (likely default; readable and lively).
- **emoji** — e.g. `🎸` (pure vibe, emoji-only).

The underlying tag is always the **text `label`**; the emoji is decoration. Even an emoji-only
chip carries a hidden text label so discovery matches on "Music", not on 🎸.

### 3.4 Scoring (personal-only, speed/rhythm-based)
Score rewards honesty and rhythm, never *which* bucket is chosen (so the tag data stays honest):
- **Reaction speed (⚡)** — a quick, decisive flick scores full points; the longer a word falls,
  the fewer points. Gut reactions are the most honest data.
- **Streak multiplier (🔥)** — a steady rhythm (no long stalls, no skips) builds ×2, ×3, …
- **Completion bonus (🏁)** — clearing the whole deck pays a lump-sum bonus.
- **All four outcomes score equally** — "Not relevant" earns the same as the seasons, so
  dismissing is never the cheap way to farm points.

### 3.5 Speed
Champion-set per round: **Slow / Medium / Fast**. Controls fall speed (and may modestly
accelerate within a round). v1 keeps it a flat per-round setting.

### 3.6 Summary (light, review-as-commit)
After completion **or** bailing mid-deck (partial sorts are kept):
- Compact score recap (final score, best streak, fastest reaction, finish bonus).
- "Your seasons, refreshed" — past / present / future chips with `new` / `moved` badges so the
  change is legible; dismissed words shown faded.
- A single primary CTA: **"Update my seasons"** applies the changes. A quiet "edit" affordance
  lets the user tweak first. No mandatory review gate. (The profile's display cap is enforced on
  the profile page, not here — see §6A; nothing is trimmed from the database.)
- Nothing writes to the profile until the user commits.

## 4. Champion / Admin Experience (seasonz-dashboard)

New section under the champion group nav: **Tag Refresh**, with three tabs.
Routes: `app/champion/[groupId]/tag-refresh/` (champion, scoped to group) and a mirror under
`app/superuser/` (admin, org-wide).

### 4.1 Build
- **Word bank panel:** a theme input (e.g. "Hobbies & interests") with **✨ Generate with AI** →
  produces N editable chips (each with a suggested emoji), each removable, plus `+ Add word` for
  manual entries. Words are season-agnostic.
- **Settings panel:** Speed (Slow/Medium/Fast segmented), Words per round (stepper), Audience
  (all members or selected groups), When (Send now; schedule deferred).
- **Push** creates a round targeting the audience and surfaces it on members' feeds.

### 4.2 Library
Saved, reusable word banks. Banks can be reused across rounds. Champion banks are scoped to
their group; admin banks can be org-wide templates.

### 4.3 Insights
Aggregated from play results (see §5.3):
- Month selector (e.g. "March 2026").
- Summary cards: rounds sent, players (e.g. 41/48), completion %, tags refreshed.
- **Most popular tags** — ranked bars, filterable by season (Past/Present/Future).
- **Movers this month** — rising and fading tags vs. the previous month.
- **Group season balance** — distribution of the group's tags across past/present/future.

### 4.4 Scoping
- **Champions** are scoped to their `group_id` for banks, rounds, and insights (enforced by RLS).
- **Admin (Seasons superuser)** pushes org-wide and sees org-level insights.

## 5. Data Model (Supabase)

### 5.1 Existing + additions
- `tags(id, label, season, category)` — unchanged.
- `user_tags(user_id, tag_id, season)` — **add columns** to support strength & history:
  ```
  created_at         timestamptz default now()
  last_reinforced_at timestamptz default now()
  strength           numeric default 0     -- decay-on-write cache (see §6A)
  reinforcement_count int default 0
  dismissed_at       timestamptz null      -- soft-dismiss; non-null = hidden from display/discovery
  ```
  The database is **never capped** — all tags (including dismissed) persist as longitudinal
  archetype data. The ~8-per-season limit is display-only (see §6A).

The commit step reuses the existing `findOrCreateTag` + `addTag` logic from
`useUserTagMutations` in the mobile app.

### 5.2 New tables
```
tag_refresh_banks
  id            uuid pk
  group_id      uuid null        -- null = org-wide / admin template
  created_by    uuid             -- author (champion or admin)
  name          text
  theme         text null
  source        text             -- 'manual' | 'ai'
  created_at    timestamptz default now()

tag_refresh_bank_words
  id            uuid pk
  bank_id       uuid fk -> tag_refresh_banks
  label         text             -- canonical text tag (always present)
  emoji         text null
  display_mode  text             -- 'text' | 'combo' | 'emoji'
  position      int
  -- NOTE: season-agnostic; the game assigns the season

tag_refresh_rounds
  id            uuid pk
  bank_id       uuid fk -> tag_refresh_banks
  group_id      uuid null        -- null = org-wide
  created_by    uuid
  speed         text             -- 'slow' | 'medium' | 'fast'
  words_per_round int
  audience      jsonb            -- 'all' | { group_ids: [...] }
  status        text             -- 'sent' | 'scheduled'
  sent_at       timestamptz null
  scheduled_for timestamptz null -- reserved; scheduling deferred
  created_at    timestamptz default now()

tag_refresh_plays
  id            uuid pk
  round_id      uuid fk -> tag_refresh_rounds
  user_id       uuid
  score         int
  best_streak   int
  fastest_ms    int
  completed     boolean
  partial       boolean
  played_at     timestamptz default now()

tag_refresh_play_words
  id            uuid pk
  play_id       uuid fk -> tag_refresh_plays
  label         text
  emoji         text null
  outcome       text             -- 'past' | 'present' | 'future' | 'not_relevant'
  reaction_ms   int null
  -- doubles as the game analytics event log (see 5.3)

tag_signals                      -- general reinforcement log across ALL engagement points
  id            uuid pk
  user_id       uuid
  tag_id        uuid fk -> tags
  season        text             -- 'past' | 'present' | 'future'
  source        text             -- 'game_sort' | 'profile_add' | 'kept' | 'post' |
                                 --   'activity' | 'discovery' | 'search' | 'dismiss'
  weight        numeric          -- source intent weight (see §6A)
  created_at    timestamptz default now()
```

### 5.3 Analytics derive from `tag_refresh_play_words`
Game-specific Insights query `tag_refresh_play_words` joined to `tag_refresh_plays` (which
carries `played_at` and the round's group):
- **Most popular tags (season, month)** = count rows where `outcome = <season>` within the month.
- **Movers** = month-over-month delta of those counts.
- **Completion / players** = aggregates over `tag_refresh_plays`.

### 5.4 Commit logic
On "Update my seasons" for a play:
- For each play-word with outcome in (past, present, future): upsert into `user_tags` via the
  existing `findOrCreateTag(label, season)` + insert (duplicate-safe), clear any `dismissed_at`,
  and write a `tag_signals` row (`source = 'game_sort'`). The `user_tags` strength cache is
  updated decay-on-write (see §6A).
- For `not_relevant`: **soft-dismiss** — set `dismissed_at = now()` on the user's tag (if present)
  and write a `tag_signals` row (`source = 'dismiss'`, negative/zero weight). The tag stays in the
  database as history; it is hidden from active display and discovery.
- Partial plays commit only the words that were sorted.

### 5.5 RLS
- Champions: read/write banks, rounds, and read insights only for their `group_id`.
- Admin: org-wide access; may create banks with `group_id = null`.
- Players: read rounds targeted to them; write their own `tag_refresh_plays` /
  `tag_refresh_play_words`; commit writes only their own `user_tags`.

## 6A. Tag Strength & Display Cap

The database stores **all** of a user's tags without limit (positive longitudinal data for
archetyping over time). The profile renders only the **strongest ~8 per season** (cap is a
configurable display constant, default 8). The user can still manually edit/pin tags.

### Strength formula
A tag is strong when reinforced **often**, **recently**, and through **higher-intent** actions:

```
strength(user, tag) = Σ  w_source · exp( −Δt / τ )
                    signals
```
- **Recency decay** — `τ` chosen for a **90-day half-life** (`τ ≈ 130 days`). This makes the
  display self-currenting, which is the whole point of the feature. A brand-new tag has full
  weight at `t = 0`, so it surfaces immediately.
- **Source weights** (`w_source`, starting defaults — all tunable):

  | Signal (`source`) | Weight |
  |---|---|
  | `game_sort` — Tag Refresh sort into a season | 1.0 |
  | `profile_add` — manually added on profile | 1.0 |
  | `kept` — survived a refresh unchanged | 0.5 |
  | `post` — used in a post / content tag | 0.4 |
  | `activity` — attached to an activity / milestone | 0.4 |
  | `discovery` — connected via tag / opened tagged content | 0.2 |
  | `search` — searched/filtered by the tag | 0.15 |
  | `dismiss` — flicked "not relevant" | 0 (zeroes display strength; sets `dismissed_at`) |

- **Display** = top-N (default 8) per season by current `strength`, excluding `dismissed_at`
  tags. Ties broken by `last_reinforced_at`.

### Computation: decay-on-write
On each new signal, decay the cached `user_tags.strength` to "now" and add the new weight:
```
strength_new = strength_old · exp( −(now − last_reinforced_at) / τ ) + w_source
last_reinforced_at = now
reinforcement_count += 1
```
This keeps profile reads cheap (just sort by `strength`) with no decay math on read. The
`tag_signals` log remains the source of truth for full recompute and cross-surface trends.

### Engagement points that emit signals
Tag Refresh commits (`game_sort`, `dismiss`) ship in this feature. Other sources (`post`,
`activity`, `discovery`, `search`, `profile_add`, `kept`) are wired opportunistically as those
surfaces are touched — the `tag_signals` schema and strength math are designed to absorb them
without change.

## 6. Components

### Mobile (Seasons_AIv02 / `src`)
- `hooks/useTagRefresh.ts` — fetch pending rounds for the current user; expose to the feed.
- A feed entry card on Discover/Home when a round is pending.
- `screens/TagRefreshGameScreen.tsx` — the playfield game loop: word spawning (sampled
  `words_per_round` from the bank), fall animation, max-two-on-screen, 4-direction flick
  detection, speed, scoring. Built on `react-native-gesture-handler` + `react-native-reanimated`.
  **(Verify both deps are already in `package.json` during planning.)**
- `components/TagRefreshSummary.tsx` — light recap + "Update my seasons" commit + edit affordance.
- Commit mutation — reuses `useUserTagMutations` (`findOrCreateTag` / `addTag`) plus a
  soft-dismiss path for `not_relevant`, writes `tag_signals` + updates the strength cache, and
  records the play + play-words.
- Profile rendering — show top-8-per-season by `strength`, excluding `dismissed_at` tags (see §6A).

### Dashboard (seasonz-dashboard)
- `app/champion/[groupId]/tag-refresh/page.tsx` + tab components: `BuildTab`, `LibraryTab`,
  `InsightsTab`.
- Admin mirror under `app/superuser/`.
- A server action / route for AI word-bank generation (theme → words + emoji) calling the
  Claude API (see §7).
- Server actions for: save bank, push round, fetch insights aggregates.

## 7. AI Word-Bank Generation
- Dashboard-side, server-only. Given a theme/category, calls the Claude API to return N words,
  each with a suggested emoji and `display_mode`.
- Use the latest appropriate Claude model; follow the `claude-api` skill at implementation time
  (prompt caching, structured output).
- Output is fully editable by the champion before saving/pushing.

---

## 8. Data Flow (end to end)
1. Champion builds/loads a bank → sets speed + audience → **Push** creates a `tag_refresh_rounds`
   row.
2. Member's app (`useTagRefresh`) sees a pending round → feed card → `TagRefreshGameScreen`.
3. Player sorts words; each sort records an intended outcome + reaction time in memory.
4. On completion/bail → `TagRefreshSummary`; **"Update my seasons"** writes `tag_refresh_plays`
   + `tag_refresh_play_words`, and commits season outcomes into `user_tags`.
5. Discovery uses the refreshed `user_tags`. Insights aggregate from `tag_refresh_play_words`.

## 9. Open Questions / Risks
- **Display cap** — *Resolved:* DB is uncapped; profile shows top-8-per-season by strength
  (default 8, configurable display constant). See §6A.
- **"Strongest tags" definition** — *Resolved:* time-decayed reinforcement score (90-day
  half-life) over weighted `tag_signals`. Weights/half-life are tunable defaults. See §6A.
- **Gesture/animation deps** — confirm `react-native-gesture-handler` + `react-native-reanimated`
  are present and the game loop performs on target devices.
- **Round re-takes** — can a user replay a round? Default: one play per round per user (v1).
- **Tuning** — source weights and the 90-day half-life are first-guess defaults; revisit once
  real engagement data exists.
